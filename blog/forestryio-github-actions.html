# 前言

今天偶然在 Nuxt Content 的 Document 中看到 Forestry.io (opens new window)，是一款免費的 Content Management System (CMS)，就來動手試試看，順便記錄一下吧！先苦後甘，現在先辛苦，以後發文就不用再碰程式碼跟 Git 了！

本篇將會介紹如何透過 Forestry.io 搭配 Github Actions 發佈 Content-based Website。

# Step 1: 註冊 Forestry.io

首先前往官網 (opens new window)註冊，註冊方式有很多種，建議使用 Github 帳號註冊，這樣就可以自動連結 Github 帳號的 Repositories。

# Step 2: 建立 site

創建帳號後會進入 Dashboard，點選右上方的「Add Site」，並在後續步驟選擇以下選項:

  1. Static Site Generator
    因為我是使用 Nuxt ，因此選擇 Others(Gridsome, Next, Nuxt ...)。
  2. Git Provider
    選擇 Github，選擇後會連結到 provider 進行授權。
  3. Repository & Branches
    由於我的 Github Pages 的 branch 分別是 main(source code) 及 gh-pages(build)，因此選擇 {username}.github.io 及 main。

# Step 3: Site settings

前往 Settings > Sidebar，點選 「Add Section」，並填入以下資料:

  1. Label
    只是一個 Label name，可以填入自己喜歡的代稱，這邊先填 Blogs。
  2. Content Directory
    Blog content 在 Github Repo 裡的相對路徑，Nuxt Content 內容是放在 content 裡，而我有對 content 作出分類 (blog/projects/...)，因此填入 content/blog。
  3. File Match
    Forestry 會自動去 repo 裡抓出對應的檔案，因此可以在這裡搭配 regex 過濾自己想要顯示及修改的檔案。由於我的 blog 都是 markdown 格式，日後文章變多也可能會往下分類，因此填入 **/*.md，即 content/blog 目錄底下不限 level 的 任意 .md 檔。
  4. Content Types
    選擇 Documents。

按下 「Done」，Forestry 便會自動到指定的路徑抓取符合的檔案，建立側邊選單並顯示目錄內容:

設定到這邊，只要你點選「Create new」就可以開始寫 blog，並且當你編輯後按下「Save」，Forestry 便會自動把你新增的 blog push 到 github 的目錄底下。

# Step 4: 新增 Github Actions

前往自己 Github Pages 的 repo ({username}.github.io)，並點選「Actions」標籤，選擇 starter-workflows ，Github 會自動創建一個 workflow template 在 repo 的 .github/workflows 目錄底下。

# Step 5: 修改 workflow

這個 start template 是一個基本的 workflow，我們需要調整及修改內容，以符合專案的架構以及 CI/CD 流程，如下:

name: CI
on:
  # 在 main branch 底下的 content 目錄發生 push **/*.md 時觸發 workflow
  push:
    branches:
      - main
    paths:
      - content/**/*.md

  # Allows you to run this workflow manually from the Actions tab
  workflow_dispatch:

jobs:
  # job 名稱為 deploy
  deploy:
    runs-on: ubuntu-latest
    steps:
    
      - name: Checkout 🛎️
        uses: actions/checkout@v2.3.1
      
      - name: Generate
        run: |
          # 安裝 package.json 裡的 dependencies
          yarn 
          # 產生 static files 到 dist 目錄
          yarn generate 
          cd dist
          # router 使用 History Mode 的解決方案
          cp index.html 404.html 
      
      # 使用 JamesIves 寫好的 Action
      # 把 static site push 到指定的 branch
      # ./dist --push-> gh-pages(branch)
      - name: Deploy 🚀
        uses: JamesIves/github-pages-deploy-action@4.1.0
        with:
          branch: gh-pages # The branch the action should deploy to.
          folder: dist # The folder the action should deploy.

由於我的專案是使用 yarn，所以在 Generate 步驟使用 yarn 安裝套件及產生 static site，如果是使用 npm 則要改成 npm install 及 npm run generate。這裡有用到 JamesIves 寫好的 Github Pages Deploy Action (opens new window)，可以指定 static site 的目錄並 deploy 到指定的 branch。

# 大功告成！

Github Pages 的後台 CMS 以及 CI/CD 流程已經建立好，現在可以馬上到 Forestry 撰寫新文章並 Save，到 Github Repo 便會看到正在進行的 Action：

到此為止，要發佈文章並自動化佈署到 Github 已經完成，但還有兩個 Forestry 的設定需要調整：

  1. 圖片路徑
    如果想在 blog post 裡用上圖片，需要調整 Forestry 裡的圖片路徑。回到 Forestry 並點選側邊欄的 「Settings」> 「Media」進行設定：

    Upload Directory
    從 Forestry 上傳的照片會被放到 Github Repo 的哪裡，這邊設定 static/uploads。

    Public Path
    當 markdown file 要插入圖片時，會到 Github Repo 的哪個路徑取得圖片。由於 Nuxt 在進行 build 時會把原本的 /static 目錄 map 到 / 目錄，因此原本上傳到 static/uploads 裡的檔案，在 build 後會被放到 /uploads 目錄，因此這邊填入 /uploads。

  2. Template
    由於 Nuxt Content 可以允許以 Front Matter 設定文章的一些 meta data，以本篇文章為例，meta data 如下:

    因此可以使用 Forestry，把每篇 blog 共通的 meta data 組成 Template。前往側邊欄的「Front matter」> 「Add Template」，這邊可以選擇直接建立新的，或是從現有的 blog 取得，相當方便： 選擇 Create based on existing document，並填入 template 名稱，以及選擇要從哪篇文章匯入，便大功告成。

以上 2 個步驟設定完，便可以直接在建立新文章時選擇 Template，快速填入meta data；也可以在文章中插入圖片，在 Github Pages 裡呈現出來！

# 心得

這是我第一次接觸第三方 CMS 及 CI/CD ，雖然我需要的設定並不複雜，但是在過程中還是處處碰壁。

例如我所有文章的建立日期都統一變成我 build 的時間，因此文章排序直接亂掉。我猜會有這樣的狀況，是因為每次執行 Github Actions 時，都會在 VM 上建立專案、重新安裝套件及產生 build，因此文件的建立時間就變成這個 timing，讓 Nuxt Content 的 createAt 屬性完全失效啊！後來解決辦法是為每篇文章加上 created 的 meta data，並小小修改 Blog component 的排序依據。
